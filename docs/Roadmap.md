# 🗺️ DataFactory 数字化工厂 - MLOps 演进路线图 (v2026.02)

> **Positioning**：本 pipeline 面向**数据采集与质检、准入决策、可追溯归档**的端到端流程，可直接复用到矿业/工业场景下的视频质检、安全巡检与感知数据质量保障（perception data curation），与 AI 驱动的安全、效率与资产利用场景对齐。

## 💡 愿景：给 AI 大脑接上高质量数据管道

人脑擅长多模态——触觉、视觉、听觉——但前提是**正确的数据**能**稳定、持续**地进入大脑，大脑才能对环境越来越熟、做出更准的决策。我们做的事本质上是同一类：把高质量的数据（经 QC、去重、不合格过滤）源源不断地喂给「大脑」，驱动它成长。

边缘计算可以类比为**眼球**：视网膜在本地先做大量预处理（亮度适应、边缘增强、运动检测），再通过视神经把已结构化的信号送给大脑，而不是原始像素。也就是说，眼球本身就是 edge computing equipment——靠近传感器、先算一层再上传。这条 pipeline 里的 edge 节点（现场质检、黄金库自检、只传摘要到中心）扮演的也是类似角色：保证送进「大脑」的是干净、可用、稳定的数据。

## 📐 完整系统结构框架（目标架构）

*主数据流：raw 数据直接进入质检，质检分为两大类别，再进入复核与归档。*

```
┌─────────────────────────────────────────────────────────────────────────────────┐
│  数据接入 (Ingest)                                                                │
│  raw_video/  [当前]     raw_lidar/  [v3 扩展]  (.pcd / .las / .ply)              │
│  可选：与 labeling 导出/回写对接 [v3]                                            │
└─────────────────────────────────────────────────────────────────────────────────┘
                                        │
                                        ▼
┌─────────────────────────────────────────────────────────────────────────────────┐
│  质检 (QC) — 两大类别，可扩展                                                      │
│                                                                                  │
│  (1) 重复检测 (Duplicate)                                                        │
│      方式：MD5 / 内容指纹 → 与 DB 已有 fingerprint 对比                           │
│      命中 → 标记为「重复」，进入复核后归 redundant_archives                      │
│                                                                                  │
│  (2) 不合格检测 (Unqualified) — 可扩展                                            │
│      当前：blur（模糊）、brightness（过曝/欠曝）、jitter（抖动）                   │
│      实现：规则/阈值 + 未来可接 YOLO 等模型（v2）                                 │
│      扩展：通过配置或插件增加新检测项（如遮挡、黑帧、分辨率异常等）[v2/v3]          │
│      任一不通过 → 标记为「不合格」，进入复核后归 rejected_material                │
│                                                                                  │
│  多模态 [v3]：视频与 LiDAR 统一走上述两类检测；LiDAR 侧增加点云质量检查           │
└─────────────────────────────────────────────────────────────────────────────────┘
                                        │
                                        ▼
┌─────────────────────────────────────────────────────────────────────────────────┐
│  复核 (Review) — 仅对「被拦」项                                                   │
│  合格 / 重复 / 不合格 汇总 → 一封邮件 → 仅对 blocked 项做 y/n/all/none 决策        │
└─────────────────────────────────────────────────────────────────────────────────┘
                                        │
                                        ▼
┌─────────────────────────────────────────────────────────────────────────────────┐
│  归档 (Archive)                                                                   │
│  qc_archive/Batch_xxx/  合格且非重复                                              │
│  rejected_material/     不合格（含 _得分pts 可溯源）                               │
│  redundant_archives/    重复（按原因归档）                                         │
└─────────────────────────────────────────────────────────────────────────────────┘
```

**Edge Deployment（v3）**：边远矿区场景下，edge 节点（现场）运行完整 pipeline，只传**结果摘要**（元数据、得分、指纹，KB 级）到中心节点，节省传输成本（GB → KB）。中心节点做汇总、复核、归档决策；需要原始数据时按需拉取。

**说明**：不合格检测采用「规则三项（blur / brightness / jitter）+ 可配置扩展」的设计，便于后续增加新检测类型（规则或模型），而不改主流程。

## 🧭 架构原则与干活节奏（干活不迷茫）

**统一架构思维**：整体框架围绕**流程**（Ingest → QC → Review → Archive），软件编排分为四类：

| 类型 | 职责 | 落地位置（目标） |
|------|------|------------------|
| **流程** | 编排「先做什么、后做什么」，不实现细节 | `core/`（ingest, qc_engine, reviewer, archiver） |
| **工具类** | 只干活、不决策（指纹、质检传感器、邮件、报告、文件操作、指标收集） | `engines/`（fingerprinter, quality_tools, notifier, report_tools, file_tools, db_tools, metrics_collector） |
| **决策类** | 根据工具输出 + 配置做判断（合格/不合格/重复、复核 y/n、归档到哪） | `core/` 中调用工具并下结论的模块，或独立 `rules/` |
| **配置类** | 工厂参数、阈值、路径、敏感信息，只存储不逻辑 | `config/`（settings.yaml, .env） |
| **指标/监控** | 系统处理指标（吞吐量、延迟、资源占用）、业务指标（合格率、重复率） | `engines/metrics_collector.py`（收集）→ DB/Prometheus（存储）→ Grafana（可视化，v3） |

**关键原则**：工具只返回数值/结果，不返回「合格/不合格」；决策层读配置 + 调工具，做判断。详见 `docs/architecture_mapping.md`（现状→目标映射表）、`docs/architecture_thinking.md`（依赖注入、接口、状态机等进阶思路）。

**版本与节奏**：推荐顺序如下；每阶段完成后再进下一阶段，避免重构与新功能并行导致混乱。

1. **v1.x**：流程跑通、人机闭环、可追溯（已完成）。
2. **v1.5**：只做架构重构（core/engines/config），不加新业务；为 v2 打牢结构（已完成）。
3. **v2.x**：在 v1.5 上加模型与实验（YOLO、版本映射、双门槛、MLflow、不合格检测可扩展）— **主体已完成**；待做：模型注册与复现；规划：多模型接入（配置/注册表 + 统一推理接口）、模型参数调整界面（做模型的人专属调参入口）。
4. **v2.5**：数据闭环与持续学习（待标池、伪标签、训练触发、新旧模型对比）；依赖 v2 双门槛与模型注册。
5. **v3.x**：高并发、多模态、云原生（状态机、Worker、Docker、Prometheus、Edge、LiDAR、labeling 对接）。
6. **v4.x**：深度溯源（Transform Log、数据血缘图谱）。

## 🚗 角色定位与协作边界（整车与发动机）

**核心比喻**：我们是造**整车**的，模型团队是造**发动机**的。一起部署到最前沿（Edge）时，整车和发动机是一辆车上的两部分，但边界要清晰：我们负责整条产线（Ingest → QC → Review → Archive）、配置、部署形态和可观测性；他们负责模型训练、验证与发布。我们**留接口**给发动机——输入（视频/帧、配置）、输出（检测结果、可选缩略图）约定清楚，模型团队要么只提供 `.pt` 满足现有推理约定，要么实现我们定义的接口（例如 `run_vision_scan(cfg, video_paths) -> List[Dict]`），即可接入，无需改我们这边的流程与报表。

**为何这样划**：若各写各的，容易重复实现（两套加载/推理）、约定不一致、部署时不知道以谁为主。由产线方定义「发动机舱」的接口，模型方实现或提供符合接口的包，部署到 Edge 时仍是**一辆车**：同一套 pipeline + 同一份配置 + 同一颗模型，责任清晰、不打架。

**职责边界与担子**：

- 我们想清楚「整车 vs 发动机 + 接口」、和模型团队一起部署到 Edge，属于**技术域内的架构与协作设计**，没有越权到 CTO 的活（战略、组织、资源、对外叙事）。这是 **MLOps / ML Platform** 该做的：把 ML 稳稳当当地跑在生产/边缘上，并定义好边界。
- **担子变重是正常的**：我们负责产线、接口、可观测性（报表、邮件、MLflow 生产 run），相当于从「只会用工具」进到「设计生产架构、定义协作契约」。这正是 Senior MLOps / ML Platform 要扛的责任。
- **职业路径**：可明确定义为 **MLOps / ML 生产 / 平台**——负责产线、接口、与模型团队一起部署到最前沿；发展上走 **Senior → Staff/Principal MLOps 或 ML Platform Lead** 即可，不必和 CTO 混为一谈。若将来想往管理/战略走，再在优先级、组织、对外上多扛一块，那是另一条路。

*（2026.02 角色与协作边界记录于此，便于与模型团队对齐、以及个人职业定位时回顾。）*

## 🔄 CI/CD 与可部署性（当前设计如何支持持续集成与持续部署）

以下设计保证「改完可验证、部署可回滚、行为可追溯」，为后续接入自动化流水线（如 GitHub Actions、Docker 部署）打基础。

| CI/CD 维度 | 当前体系中的实现 | 作用 |
|------------|------------------|------|
| **配置与代码分离** | 行为由 `config/settings.yaml` + `.env` 驱动，`config_loader` 统一加载 | 改配置即可换环境/阈值/模型，无需改代码；部署 = 换配置 + 重启 |
| **单一入口** | `main.py`（单次 / `--guard`）唯一入口 | 集成结果就是「这一条命令」，便于测试与部署 |
| **启动自检** | `startup_self_check`：校验路径存在且可写 | 配置错或目录错立刻失败，不悄悄跑错 |
| **黄金库自检（可选）** | `startup_golden_run`：用 golden 视频真跑一遍 QC | 部署后冒烟：确认整条流水线能跑通再接真实流量 |
| **冒烟测试** | `scripts/smoke_test.py`：造测试物料 + 跑 QC + 断言 | 集成测试：每次改代码/配置都可跑一遍，保证主流程不断 |
| **环境可重置** | `scripts/reset_factory.py`（dry-run / --execute） | 测试前可清空 test/raw，保证可重复的测试状态 |
| **版本可追溯** | M2 的 `version_mapping`、`version_info.json`、path_info | 每一批都知道用的哪版规则/模型；出问题可回滚、可审计 |
| **环境一致** | `init_storage_structure` + `requirements.txt` + venv | 目录与依赖固定，不同机器跑出来行为一致 |

**尚未自动化**：提交代码后自动跑 `smoke_test`（需 CI 如 GitHub Actions）；标准化部署载体（v3 的 Docker）与「换配置/换模型后自动重启 Guard」的 CD 脚本。当前是「设计已支持 CI/CD，自动化流水线待接」。

## 📈 规模扩大与架构演进：做公司的「神经系统」

**定位**：在 AI 时代，本系统希望成为公司的**眼睛、皮肤、鼻子**——即通向「大脑」（决策、训练、业务）的**感知与过滤层**。如同生物体：感官在边缘先做大量预处理，只把有意义的结构化信号传给大脑，而不是原始海量数据。

**尤其面对偏远矿区等场景**：若原始视频/传感器数据全量回传，带宽成本极高且多数数据价值有限。系统像**眼球**一样在边缘完成 Ingest → QC（规则 + 可选模型），只回传**结果摘要**（元数据、得分、指纹、QC 报告，KB 级），中心做汇总、复核与归档决策；需要原始数据时再按需拉取。这样从 GB 级回传压到 KB 级，成本与延迟都可控。

**底层架构是否需要推倒重来？** 不需要。四步骨架（Ingest → QC → Review → Archive）是稳定的；规模扩大时是**同一套流程、更多节点与角色**：

| 阶段 | 形态 | 说明 |
|------|------|------|
| **单机** | 一台跑 `main.py` / `--guard` | 当前形态；配置化、可追溯、可测 |
| **多 Worker（v3）** | 消息队列 + 多进程/多机并行质检 | 仍是四步；只是 Ingest 后任务分发、QC/Review/Archive 可水平扩展 |
| **Edge + 中心** | 矿区/现场跑完整四步，只上传摘要；中心汇总、复核、归档 | 同一套流程在 Edge 跑一遍，输出从「原始数据」变为「摘要」再回传；配置与模型可下发，Edge 轻量化（v3） |

因此：**底层是「流程 + 配置 + 版本」；扩展时加的是节点数、网络拓扑和「谁算、谁汇总」，而不是重写核心逻辑。** 这样既能在公司变大的时候平滑扩展，又能持续扮演「感官 + 边缘预计算」的角色，把有意义的信号送进「大脑」，控制带宽与成本。

## 🔀 多节点部署：避免冲突、协同与中心汇总分发

**问题**：若同一套系统部署在多个节点甚至上百个节点（如各矿区、各车间），**不做设计就会冲突、打架**：批次号重复、DB 主键冲突、同一份原始数据被多节点重复处理、中心无法区分来源与时间顺序。

**设计目标**：  
（1）**无冲突**：不同节点产生的批次、记录、文件在全局唯一、可区分。  
（2）**可协同**：要么各节点负责不同数据源（按站点/产线划分），要么由中心/队列分配任务，避免重复处理。  
（3）**中心可整理**：云上中心接收各节点回传的摘要/元数据，统一入库、去重、按批次/节点/时间组织，便于分发给后续团队（标注、训练、分析等）。

---

### 1. 冲突从哪里来、如何避免

| 冲突类型 | 原因 | 应对 |
|----------|------|------|
| **批次号碰撞** | 当前 `batch_id = 时间戳`（如 `%Y%m%d_%H%M%S`），多节点同一秒会重名 | 引入 **node_id**（节点唯一标识，配置或环境变量）；`batch_id = node_id + "_" + 本地时间戳`（或 UUID），保证全局唯一 |
| **DB 主键冲突** | 若多节点写同一中心 DB，`batch_id` 主键会冲突 | 同上，batch_id 含 node_id 即全局唯一；或各节点写本地 DB，中心只收摘要不直接写节点 DB |
| **目录/文件冲突** | `Batch_xxx`、`Batch_xxx_Fails` 若多节点同号会覆盖 | batch_id 含 node_id 后，目录名自然区分（如 `Batch_node01_20260220120000`） |
| **重复处理同一份数据** | 多节点从同一 raw 源拉取，会重复质检 | **数据源划分**：每节点只消费指定站点/目录/队列；或 **中心下发任务**：由消息队列/任务表分配「这一批由 node_id=X 处理」，其它节点不碰 |

实现要点：**配置（或环境）中必含 `node_id`**；所有上报、日志、写入中心的数据都带 `node_id`；若用中心 DB，主键或联合主键含 `(node_id, batch_id)` 或直接用「node_id + 本地 batch_id」作为全局 batch_id。

---

### 2. 如何协同工作

- **按源划分（推荐用于地理分散的 Edge）**：每个节点只处理本地的 raw（本矿区的 storage/raw），互不重叠。协同仅体现在「都向同一中心上报摘要」，中心不做任务分配，只做汇总。
- **中心分配任务（用于共享数据源时）**：中心或队列维护「待处理批次/任务」；节点拉取任务时带上 `node_id`，将任务标记为「已被 node_id 领取」，处理完后上报结果并标记完成。这样同一批数据只会被一个节点处理。
- **配置与模型下发**：中心可向各节点下发统一或按节点定制的配置（阈值、模型路径、开关），保证行为一致或可灰度；v3 Edge 同步机制（结果上传、配置下发）即为此准备。

---

### 3. 中心如何整理回传数据并分发给后续团队

- **接收**：各节点按约定协议（如 HTTP/消息队列）上传**结果摘要**：`node_id`、`batch_id`、文件列表、得分、指纹、QC 报告 JSON、版本映射等（KB 级，不含原始视频）。
- **入库与规范化**：云服务器将每条摘要写入**中心库**（如关系表或时序/文档库），必含字段：`node_id`、`batch_id`（全局唯一）、时间、文件数、得分分布、指纹列表、version_info 等；可选：原始文件路径（若按需拉取则后填）。
- **去重与关联**：按指纹（或业务主键）做**跨节点去重**，避免同一内容被多节点重复计入；需保留「来自哪几个 node、哪几个 batch」时可保留明细，汇总层做聚合。
- **分发给后续团队**：  
  - **标注团队**：从中心库按「待标池」条件筛选（如低置信、未标、某时间段），导出 manifest 或通过 API 提供给 Label Studio/CVAT。  
  - **训练/算法团队**：按批次、节点、时间范围导出元数据与 QC 结果，或开放只读视图/API，便于做数据闭环与模型迭代。  
  - **运营/分析**：按 node_id、时间、通过率等做报表与监控。

这样：**多节点不打架（node_id + 唯一 batch_id）、协同清晰（按源或按任务）、中心统一整理并可按需分发给标注/训练/运营等后续团队。**

---

### 4. 与当前代码的关系

- 当前 `batch_id` 为本地时间戳，且无 `node_id`，**仅适用于单节点**。多节点改造时需在配置/环境中增加 `node_id`，并在生成 `batch_id`、写入 DB、上报中心时统一使用「node_id + 本地 batch_id」或等价全局 ID；`sync_id`（DB 已预留）可用于与中心同步时的关联。  
- v3 的「Edge 只传摘要、中心汇总」与本节一致；本节把「冲突避免、协同、中心整理与分发」显式化，便于多节点/上百节点时的落地。**具体实现项**见阶段三 (v3.x) 的 **多节点部署（实现项）**，可后续慢慢往里面加。

## 🧠 阶段二：视觉感知与自动化准入 (v2.x)

- [x] **计算机视觉质检接入**：YOLO 单例加载、按 config 抽帧与推理（conf/iou/classes/device 等全配置化）；仅返回检测结果，决策在 qc_engine。

- [x] **不合格检测可扩展**：`quality_tools.register_extra_check` 注册额外检测项，`decide_env` 统一调度；便于接入黑帧、分辨率异常等。

**模型参数调整界面（规划）**：给「做模型的人」专属调参入口，避免其改 pipeline 或整份 `settings.yaml`。可选落地方式：
- **按模型分片配置**：每个模型在 config 中独立一块（如 `vision_models.truck_detector`、`vision_models.material_classifier`），内含 `model_path`、`conf`、`iou`、`device`、`imgsz` 等；做模型的人只编辑自己负责的那一段配置（或通过下文界面间接修改）。
- **模型参数配置页/API**：提供小型 Web 页或 CLI/API，仅暴露「模型选择 + 推理参数」（conf、iou、device、imgsz、sample_seconds 等）；保存时写入 config 或独立 `config/model_params.yaml` / DB，pipeline 读取后用于推理。权限上可限制为仅能改「自己名下的模型」对应参数。
- **与 MLflow 联动（可选）**：若模型注册在 MLflow，可将「推荐推理参数」写在 Model Registry 的 metadata 或 run params；pipeline 拉取模型时一并读取推荐参数，并允许本地 config 覆盖，做模型的人在 MLflow UI 或上述配置页中维护均可。

目标：做模型的人只接触「自己的模型 + 自己的参数」，不动 pipeline、不碰其它模型配置。

## 🔄 阶段二点五：数据闭环与持续学习 (v2.5)
*核心：在 v2 具备「模型推理 + 双门槛 + 模型注册」后，形成「新数据 → 按置信度分流 → 待标池/伪标签 → 训练触发 → 新旧模型对比」的闭环；依赖 v2，先于 v3 规模扩展。*

- [ ] **按置信度自动分流**：与 v2 双门槛衔接；高分自动放行、低分自动拦截、中间态自动进入待标池（或人工复核队列）。
- [ ] **按置信区间导出 / 重命名批次产出结构**：将 `2_Mass_Production` 改为按置信分层落盘——高置信=燃料、中间=人工、低=报废；详见 `docs/batch_output_confidence_tiers.md`。实现上可先改命名与文档，再迭代 pipeline 落盘逻辑。
- [ ] **自动生成待标池**：低置信度/不确定样本自动写入待标池（如 `storage/for_labeling` + manifest），与现有 `export_for_labeling` 对齐或扩展；支持按批次或按阈值筛选。
- [ ] **自动生成伪标签（可选策略）**：高置信度预测可写为伪标签，供训练或人工抽检；策略可配置（阈值、类别、是否需人工确认）。
- [ ] **标注回写与训练触发**：标注结果回写 DB，与批次/模型版本关联；新标注积累到一定量或定时触发训练任务（或触发「训练就绪」事件，由外部调度执行）。
- [ ] **新旧模型对比**：新模型训练完成后，与当前线上/注册模型做离线或在线对比（指标、A/B）；结果写入 MLflow 或 DB，支持上线决策。

**完成标准**：新数据经模型推理后可按置信度进入不同路径；待标池可自动更新；训练触发与模型对比有明确入口与记录。

## 🐳 阶段三：高并发、多模态与云原生架构 (v3.x)
*核心：海量数据吞吐、多传感器支持（LiDAR）、任务编排与部署、与 labeling 工作流对接（含 on-edge 方向）*

**任务编排与扩展能力**

- [ ] **任务状态机 (State Machine)**：数据库引入 `Pending/Processing/Reviewing/Done/Fail`，支撑多任务调度与监控。
- [ ] **多进程/分布式 Worker**：消息队列 + 多视频（及多模态任务）并行质检，消除单机 I/O 瓶颈，支持大规模数据管道。
- [ ] **容器化部署 (Docker)**：环境一键迁移；结合 Prometheus 监控资源占用，为生产与 on-edge 部署打基础。
- [ ] **系统处理指标监控（Prometheus + Grafana）**：
    - **吞吐量指标**：每小时处理 GB 数、文件数、批次完成数；各阶段（Ingest/QC/Review/Archive）的吞吐量。
    - **延迟指标**：单文件处理时间（P50/P95/P99）、批次总耗时、各阶段耗时分布。
    - **资源占用**：CPU、内存、磁盘 I/O、GPU（若用 YOLO）使用率；队列长度、Worker 利用率。
    - **业务指标**：合格率、重复率、人工复核率、归档分布（合格/废片/冗余）。
    - **指标位置**：`engines/metrics_collector.py`（收集）、`core/` 各模块调用收集器、Prometheus 暴露端点、Grafana Dashboard 可视化。
    - **用途**：面试展示（「系统每小时处理 X GB，P95 延迟 Y 秒」）、生产监控（资源告警、性能优化依据）。
- [ ] **Edge Deployment（边缘部署）— 节省传输成本**：
    - **场景**：边远矿区带宽有限，传原始视频（GB 级）成本高；在 edge 节点（矿区现场）直接处理，只传**处理结果**（元数据、质检报告、指纹、得分，KB 级），显著节省传输成本。
    - **架构**：
      - **Edge 节点**（矿区现场）：运行完整 pipeline（Ingest → QC → Review），本地存储合格/不合格/重复的元数据；只传**结果摘要**（Batch_ID、文件列表、得分、指纹、QC 报告 JSON）到中心节点。
      - **中心节点**（公司总部）：接收 edge 结果，做**汇总、复核、归档决策**；需要原始数据时，edge 节点按需上传（如「被拦项需要人工复核，上传原始视频片段」）。
    - **技术实现**：
      - **模型轻量化**：YOLO 量化/剪枝，适配 edge 设备（Jetson、工控机、车载平台）；规则质检（blur/brightness/jitter）本身轻量，edge 友好。
      - **同步机制**：edge ↔ 中心双向同步（结果上传、配置下发、模型更新）；断网时 edge 本地缓存，联网后批量同步。
      - **数据策略**：edge 本地保留原始数据（短期，如 7 天），中心节点只存元数据和摘要；需要原始数据时按需拉取。
    - **商业价值**：节省传输成本（GB → KB）、降低延迟（本地处理）、支持离线运行（断网可用）；与 LoopX 的 "deploy/monitor models on embedded or vehicle platforms operating in harsh environments" 高度对齐。

**多节点部署（实现项）**  
*同一套程序，中心与各节点通过配置区分；设计见上文「多节点部署：避免冲突、协同与中心汇总分发」。以下为可逐步落地的实现项。*

- [ ] **node_id 与全局 batch_id**：在 config/settings.yaml 或环境中支持 `node_id`（节点唯一标识）。**若未配置 node_id，则自动取本机 MAC 地址或主机名**，防止运维未改配置导致多机同名（如都叫 node01）；`batch_id` 生成改为 `node_id + "_" + 本地时间戳`（或 UUID），保证多节点无碰撞；目录名、DB 主键等沿用该全局 batch_id。
- [ ] **节点侧：上报摘要**：批次结束后，将摘要（node_id、batch_id、文件列表、得分、指纹、version_info、QC 报告路径或 JSON）按约定协议上报中心（如 HTTP API 或消息队列）。
- [ ] **摘要回传可靠性 — 本地重试（Store-and-Forward / 断点续传）**：断网时摘要先落本地数据库，联网后自动批量补传；未成功上报的批次在本地标记，定时重试直至中心确认。工业级标配，保证不丢摘要。
- [ ] **中心侧：接收与入库**：中心提供接收接口，将各节点上报的摘要写入中心库（表结构含 node_id、batch_id、时间、文件数、得分、指纹列表等）；支持按指纹跨节点去重与关联。
- [ ] **中心侧：分发出口**：按批次/节点/时间/条件查询中心库，导出待标池 manifest 或开放 API，供标注团队、训练团队、运营分析使用。
- [ ] **配置与部署说明**：文档明确「主节点配主节点参数、各节点配各节点参数」（node_id、路径、中心 URL、是否上报等）；同一套代码多份部署，每份带各自配置即可。

**多模态数据支持（LiDAR）**

- [ ] **LiDAR 点云数据接入**：
    - 支持常见点云格式（`.pcd`, `.las`, `.ply`），与 `raw_video/` 同级增加 `raw_lidar/` 或统一入口。
    - 点云质量检查：点数、密度、异常空洞、坐标系等；与视频质检并列，走同一套「重复检测 + 不合格检测」逻辑。**点云特殊质检**：预留 **spatial_consistency（空间一致性）** 接口，针对点云最怕的「空洞」与「坐标漂移」做不合格检测，便于后续接入具体算法。
    - 与视频的**时间戳对齐**（timestamp matching）：同一 Batch 内 video ↔ lidar 配对，为后续 sensor fusion 打基础。
- [ ] **统一多模态质检**：重复检测（指纹/哈希）与不合格检测在视频与 LiDAR 两侧统一策略；归档结构支持 `Batch_xxx/video/` 与 `Batch_xxx/lidar/` 对齐。

**Labeling 工作流集成**

- [ ] **与标注流程对接**：
    - 从 pipeline 导出「待标注」数据到 labeling 工具（如 Label Studio / CVAT），避免实习生重复标注已去重数据。
    - 标注完成后回写：标注文件与原始数据在 DB 中关联，记录「哪批标注对应哪次训练/哪个模型版本」，纳入数据血缘（v4）。

**质检扩展（与框架一致）**

- [ ] **不合格检测扩展**：在 v2 可配置/插件基础上，多模态下支持 LiDAR 侧的不合格规则（如点云密度不足、范围异常）；**LiDAR 侧预留 spatial_consistency（空间一致性）**，用于空洞、坐标漂移等检测。保持「重复 + 不合格」两大类的清晰划分，便于后续继续加检测项。

## 🚛 场景扩展：矿车关键帧提取与物料识别（日后实现）

*以下为讨论结论记录，供日后实现时对照。*

**业务目标**：顶视固定机位拍摄矿车经过，只需**一帧**——车中心经过画面中心的那一帧。该帧用于：(1) 与 Lidar 在该时刻扫车斗 bbox 的体积对齐，反推产量；(2) 后续用模型识别装载物类型（砂石/矿石/煤矿/其他）。

**关键帧选取（A）**  
- 定义：车中心（bbox 中心）与画面中心距离 **最小** 的那一帧。  
- 流程：抽帧（降采样）→ 每帧得到矿车 bbox → 每段「连续有车」内取 \(d = \|P - C\|\) 最小的帧 → 输出关键帧 + 时间戳 \(t\)。  
- 可复现：固定抽帧间隔、固定选帧规则（可加 bbox 中心平滑）。

**bbox 获取：先数学后模型**  
- **阶段一（无模型）**：用背景减除（或帧差）+ 轮廓 → 前景连通块外接矩形 = 粗 bbox；用该 bbox 中心按上述规则挑关键帧。得到大批「关键帧 + 粗 bbox」。  
- **阶段二**：这批关键帧 + 粗 bbox 作为初版标注（可人工抽检/修正），训练顶视矿车检测模型（如 YOLO）。  
- **阶段三**：新视频用训练好的模型出 bbox，再按「车中心过画面中心」选关键帧；Lidar 用时间 \(t\) 与 bbox 对齐做体积/产量（B）。

**产出**：每辆车一条记录 = 关键帧（图像）+ 时间 \(t\) + bbox（车斗/整车视需求）；该帧后续进物料分类模型（砂石/矿石/煤矿/其他）。
